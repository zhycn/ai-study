# 进阶篇：复杂推理与多步策略

掌握基础提示词技巧后，本章将深入讲解如何引导大语言模型进行复杂推理、多步决策和自我优化。这些进阶技巧是构建高质量 AI 应用的核心能力。

## 思维链（Chain of Thought，CoT）

### Zero-shot CoT

**零样本思维链（Zero-shot Chain of Thought）**是指不提供推理示例，直接通过触发语引导模型逐步思考的技术。这是最简单也最实用的 CoT 形式。

2022 年 Google 的研究发现，仅通过在提示末尾添加"让我们逐步思考"（Let's think step by step），大语言模型在 GSM8K 数学基准测试上的准确率就从 17.9% 提升到 54.3%。其原理是触发模型生成中间推理步骤，而非直接跳到结论。

```text
请逐步分析以下问题：
1. 首先理解问题的核心要求
2. 列出已知条件
3. 推导中间步骤
4. 得出最终结论

问题：如果一个项目需要 3 个人 4 天完成，那么 6 个人需要几天？
```

::: tip 提示
Zero-shot CoT 的常见触发语包括："让我们逐步思考"、"请逐步解释你的推理过程"、"先分析再给出答案"。不同触发语在不同模型上的效果可能有差异，建议针对目标模型单独测试。
:::

### Few-shot CoT

**少样本思维链（Few-shot Chain of Thought）**通过提供包含完整推理过程的示例，让模型学习"如何思考"。相比 Zero-shot，Few-shot CoT 能更精确地控制推理的格式和深度。

```text
问题：小明有 5 个苹果，吃了 2 个，又买了 3 个，现在有几个？
解答：
- 小明开始有 5 个苹果
- 吃了 2 个，剩下 5 - 2 = 3 个
- 又买了 3 个，现在有 3 + 3 = 6 个
答案：6 个

问题：一个水池有 100 升水，每小时流出 5 升，多少小时后剩 50 升？
解答：
```

Few-shot CoT 的关键在于示例质量。示例应覆盖不同难度级别，推理步骤要清晰完整，最终答案的格式要与目标任务一致。

### 适用场景与效果对比

思维链特别适用于以下场景：

- **数学计算**：多步运算、方程求解、几何证明
- **逻辑推理**：条件判断、演绎推理、谜题解答
- **复杂分析任务**：代码调试、故障排查、根因分析
- **决策支持**：方案评估、风险识别、利弊权衡

| 方式          | 准确率 | 适用场景   | Token 消耗 |
| ------------- | ------ | ---------- | ---------- |
| 直接回答      | 低     | 简单问答   | 少         |
| Zero-shot CoT | 中     | 中等复杂度 | 中         |
| Few-shot CoT  | 高     | 复杂推理   | 多         |

::: warning 注意
CoT 会显著增加 Token 消耗，因为模型需要生成完整的推理过程。在成本敏感的场景中，可以考虑仅在复杂任务中启用 CoT，或在获取推理后仅向用户展示最终答案。
:::

### 最佳实践

- 调试提示词时先将**温度（Temperature）**设为 0，确保推理逻辑稳定后再调整
- 推理步骤的粒度要适中，过细会浪费 Token，过粗则失去 CoT 的意义
- 对于代码调试类任务，引导模型按照"意图 → 实际行为 → 差异 → 修复方案"的结构分析
- 可以结合 [RAG](/glossary/rag) 使用，先检索相关知识再生成推理链

## 思维树（Tree of Thoughts，ToT）

### 概念

**思维树（Tree of Thoughts，ToT）**是思维链的扩展，让模型在推理过程中探索多条路径，自行评估和选择最优解。如果说 CoT 是一条线性的思考路径，那么 ToT 就是一棵分叉的思考树。

ToT 的核心思想来源于人类的决策过程：面对复杂问题时，我们通常会考虑多种可能性，逐一评估后再做出选择。ToT 将这一过程形式化为四个阶段。

### 工作流程

1. **生成（Thought Generation）**：针对当前状态，产生多个可能的思考分支
2. **评估（State Evaluation）**：对每个分支的可行性和进展程度进行评估打分
3. **选择（Search Algorithm）**：使用搜索算法（如广度优先搜索、深度优先搜索）选择最优路径继续
4. **迭代（Iteration）**：重复生成、评估、选择，直到得出结论或达到最大深度

```text
请针对以下问题探索多种解决方案：

问题：如何优化一个响应时间 3 秒的 API 接口？

请按以下步骤：
1. 列出 3 种可能的优化方向（例如：数据库优化、缓存引入、异步处理）
2. 对每种方向评估：
   - 可行性（高/中/低）
   - 预期效果（响应时间可降低多少）
   - 实施成本（开发工作量）
3. 综合评估后选择最优方案，并详细说明实施步骤
```

### 适用场景

- **需要多方案比较**：技术选型、架构设计、方案评审
- **创意发散**：产品创意、营销策略、内容策划
- **复杂决策**：资源分配、优先级排序、风险评估
- **数学难题**：需要尝试多种解法才能找到正确答案的问题

::: info 信息
ToT 相比 CoT 的主要优势在于能够"回溯"——当某条推理路径走入死胡同时，可以回到上一个分支点尝试其他路径。这种能力在解决复杂数学谜题和编程难题时尤为有用。
:::

### 最佳实践

- 明确指定生成分支的数量（通常 3-5 个），避免模型产生过多无效分支
- 为评估阶段提供明确的评分维度，如可行性、成本、效果等
- 对于特别复杂的问题，可以限制搜索深度，防止 Token 消耗过大
- 结合代码实现时，可以使用程序控制搜索流程，而非完全依赖模型自主决策

## 自我反思与迭代（Self-Refinement）

### 概念

**自我反思与迭代（Self-Refinement）**让模型在生成内容后，自行审查、发现问题并修正输出。其核心循环是：**生成 → 审查 → 修正**。

人类在写作或编程时通常会先完成初稿，再反复修改。Self-Refinement 将这一过程引入提示词工程，通过多轮交互持续提升输出质量。研究表明，经过一轮自我修正后，模型输出的准确率通常可以提升 10-20%。

### 标准模板

```text
# 第一步：生成初稿
请撰写一篇关于{主题}的技术文章。

# 第二步：自我审查
请检查上面的文章，回答以下问题：
- 技术描述是否准确？有没有过时或错误的信息？
- 是否有逻辑漏洞或论证不充分的地方？
- 是否有表述不清、读者可能产生误解的地方？
- 结构是否合理？各章节之间的过渡是否自然？

# 第三步：修正输出
根据上述审查结果，输出修改后的完整文章。标注所有修改之处并说明修改原因。
```

### 示例：代码审查

```text
请审查以下代码，找出潜在问题：

{代码片段}

审查后，请完成以下任务：
1. 列出所有发现的问题，按严重程度排序
2. 解释每个问题的风险和可能导致的后果
3. 给出修正后的完整代码
4. 说明修正前后的差异和改进点
```

::: tip 提示
自我反思可以多次迭代。对于高质量要求的内容，可以设置"生成 → 审查 → 修正 → 再审查 → 再修正"的多轮循环。但要注意每次迭代都会增加 Token 消耗和响应时间。
:::

### 适用场景

- **代码审查**：发现潜在 bug、性能瓶颈、安全隐患
- **文章润色**：修正逻辑错误、优化表述、统一风格
- **方案优化**：识别设计缺陷、补充遗漏点、提升可执行性
- **数据校验**：检查数据一致性、发现异常值

### 最佳实践

- 审查阶段的问题要具体明确，避免"你觉得怎么样"这类模糊提问
- 可以引入外部标准（如编码规范、写作指南）作为审查依据
- 对于关键任务，可以结合 [自一致性](#自一致性-self-consistency) 使用，先生成多个版本再交叉审查
- 修正阶段要求模型标注修改内容，便于人工复核

## ReAct 模式

### 概念

**ReAct（Reasoning + Acting）**模式将推理（Reasoning）与行动（Acting）相结合，形成"思考 → 行动 → 观察"的循环。这是 AI [Agent](/glossary/agent) 的核心行为逻辑。

传统提示词中，模型只能基于训练数据中的知识回答。ReAct 模式让模型能够与外部世界交互：通过工具获取实时信息，基于观察结果调整思考方向，最终给出更准确的答案。

### 工作流程

ReAct 的标准循环包含三个步骤：

1. **Thought（思考）**：分析当前状态，决定下一步行动
2. **Action（行动）**：调用工具执行操作（搜索、查询 API、运行代码等）
3. **Observation（观察）**：获取工具返回的结果，作为下一步思考的依据

```text
Thought: 我需要先搜索相关信息
Action: 搜索[关键词]
Observation: [搜索结果]
Thought: 根据搜索结果，我需要进一步...
Action: ...
```

### 示例

```text
你是一个研究助手，可以使用搜索工具。请按以下格式回答：

Thought: 我需要了解 Transformer 的起源
Action: 搜索"Transformer 论文 2017"
Observation: Attention Is All You Need, Vaswani et al., NeurIPS 2017
Thought: 现在我需要总结该论文的核心创新点
Action: 分析论文创新点
Observation: 自注意力机制、位置编码、并行计算架构
Thought: 我还需要了解后续影响
Action: 搜索"Transformer 后续发展 BERT GPT"
Observation: BERT(2018)、GPT(2018)、T5(2019) 等衍生模型

最终回答：Transformer 架构由 Vaswani 等人在 2017 年提出...
```

::: warning 注意
ReAct 模式需要外部工具支持。在纯文本对话中，模型无法真正执行"行动"步骤，只能模拟该过程。要实现完整的 ReAct 循环，需要结合 [工具使用](/glossary/tool-use) 框架（如 LangChain、LlamaIndex）。
:::

### 适用场景

- **需要外部工具调用**：搜索、数据库查询、API 调用、代码执行
- **信息检索 + 分析**：需要实时数据支持的分析任务
- **多步骤任务**：需要按顺序执行多个操作才能完成的目标
- **动态决策**：根据中间结果调整后续行动的场景

### 最佳实践

- 明确定义可用的工具列表及其功能描述，这本质上是结构化的 [函数调用](/glossary/function-calling)
- 为每个 Thought 步骤设置清晰的决策依据，避免盲目行动
- 观察结果可能包含噪声，引导模型学会筛选有效信息
- 设置最大迭代次数，防止模型陷入无限循环

## 提示词链设计

### 概念

**提示词链（Prompt Chaining）**将复杂任务拆分为多个子任务，每个子任务使用独立的提示词处理，上下文在链中逐步传递。这与软件工程中"单一职责原则"的思想一致——每个提示词只做好一件事。

提示词链的优势在于：降低单个提示词的复杂度、提高每步输出的质量、便于定位和修复问题、支持人工介入审核中间结果。

### 示例：研究写作链

```text
步骤 1：信息收集
提示词：请列出关于{主题}的 5 个关键问题，覆盖技术原理、应用场景、优缺点、发展趋势和常见误区。

步骤 2：大纲生成
提示词：根据以下关键问题，生成一篇技术文章的大纲，包含引言、正文（分章节）和结论。
输入：{步骤 1 的输出}

步骤 3：内容撰写
提示词：根据以下大纲，撰写第一章的详细内容。要求技术准确、逻辑清晰、配有代码示例。
输入：{步骤 2 的输出}

步骤 4：审校优化
提示词：请审查并优化以下文章，检查技术准确性、逻辑连贯性和语言表达。
输入：{步骤 3 的输出}
```

### 上下文传递策略

在提示词链中，上下文传递方式直接影响最终质量：

- **全量传递**：将上一步的完整输出作为下一步输入。适合信息量不大的场景
- **摘要传递**：先对上一步输出进行摘要，再传递给下一步。适合长文本场景，可节省 Token
- **关键信息提取**：只提取上一步中与当前任务相关的信息。适合复杂场景，减少干扰
- **约束继承**：保留全局约束条件（如语言风格、格式要求）在每一步中重复强调

::: tip 提示
在设计提示词链时，建议在关键节点设置人工审核点。例如在大纲生成后让人工确认方向正确，再继续内容撰写。这种 **人在回路（Human-in-the-loop）** 的方式可以显著降低最终输出的返工率。
:::

### 适用场景

- **长文写作**：研究论文、技术博客、产品文档
- **研究报告**：行业分析、竞品调研、技术评估
- **复杂分析**：多因素决策、风险评估、方案对比
- **数据处理流水线**：数据清洗 → 特征提取 → 分析建模 → 结果可视化

### 最佳实践

- 每个节点的提示词应独立可测试，便于单独优化
- 在链的入口处定义全局约束，并在关键节点重复强调
- 为每个节点设置明确的输出格式要求，便于程序化处理
- 记录每一步的输入输出，便于回溯和调试

## 自一致性（Self-Consistency）

### 概念

**自一致性（Self-Consistency）**通过对同一问题生成多条推理路径，然后通过投票或一致性检查选择最可靠的答案。其核心假设是：正确的推理路径往往会产生相同的答案，而错误的路径则倾向于给出不同的结果。

自一致性与 [思维链](/glossary/chain-of-thought) 密切相关。它不是改变单次推理的方式，而是通过多次采样增加找到正确推理路径的概率。

### 实现方式

```text
请对以下问题给出 3 种不同的分析角度：

问题：{问题描述}

角度 1（从技术实现角度）：
角度 2（从用户体验角度）：
角度 3（从商业价值角度）：

请比较 3 种角度的分析，找出最合理的结论。
```

在程序层面，实现自一致性通常包括以下步骤：

1. 使用非零温度（如 0.7）生成多条不同的推理路径
2. 从每条推理中提取最终答案
3. 统计各答案的出现频率
4. 选择出现次数最多的答案作为最终输出

### 成本与收益

| 维度       | 单次生成 | 自一致性（3 次） |
| ---------- | -------- | ---------------- |
| Token 消耗 | 1x       | 3x               |
| 准确率     | 基准     | 提升 10-20%      |
| 响应时间   | 基准     | 约 3 倍          |
| 适用场景   | 日常任务 | 高可靠性要求     |

::: warning 注意
自一致性的成本与生成次数成正比。在实际应用中，通常 3-5 次采样即可取得显著效果提升，继续增加次数边际效益递减。
:::

### 适用场景

- **关键决策**：影响重大的技术选型、架构设计
- **高准确率要求**：医疗诊断辅助、法律咨询、财务分析
- **容错率低的场景**：安全审查、合规检查、代码发布前验证
- **答案可验证**：数学计算、逻辑推理等有明确正确答案的任务

### 最佳实践

- 设置合适的温度（0.5-0.8），保证生成结果的多样性
- 答案提取逻辑要精确，避免因格式差异导致同一答案被识别为不同结果
- 对于开放性任务（如创意写作），自一致性效果有限，更适合封闭性任务
- 可以结合置信度阈值使用，当最高票答案的得票率低于阈值时触发人工审核

## 与其他概念的关系

- [思维链](/glossary/chain-of-thought) 是 CoT 和 ToT 的基础，所有进阶推理技巧都建立在逐步思考的能力之上
- [Agent](/glossary/agent) 的行为逻辑依赖 ReAct 模式，思考与行动的循环是智能体的核心工作流
- [工具使用](/glossary/tool-use) 是 ReAct 中"行动"环节的延伸，决定了 Agent 能与哪些外部系统交互
- [规划（Planning）](/glossary/planning) 能力通过提示词链实现，将大目标分解为可执行的子任务序列
- [人在回路（Human-in-the-loop）](/glossary/human-in-the-loop) 在提示词链的关键节点引入人工审核，提升整体可靠性
- [工作流（Workflow）](/glossary/workflow) 是提示词链的工程化实现，将多个提示词节点编排为可执行的流水线
