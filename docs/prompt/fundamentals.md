# 基础篇：LLM 原理与提示词基础

理解大语言模型（Large Language Model，LLM）的工作原理是编写高效提示词（Prompt）的前提。本章将从 Token 化、上下文窗口、温度参数等底层机制出发，系统讲解提示词工程的基础概念与实用技巧。

## LLM 如何理解提示词

### Token 化过程

**Token（词元）**是模型处理文本的基本单位。模型并不直接"理解"人类语言，而是将文本拆解为 Token 序列，再通过数学运算生成输出。

中英文 Token 化存在显著差异：

- **英文**：通常以单词或子词（Subword）为单位，如 "unbelievable" 可能被拆分为 "un" + "believ" + "able" 三个 Token
- **中文**：通常以字或词为单位，如 "人工智能" 可能被拆分为 "人工" + "智能" 两个 Token，也可能按单字拆分为四个 Token

Token 数量直接影响使用成本。大多数 API 按 Token 数量计费，输入和输出分别计算。了解 Token 化机制有助于：

- 估算提示词成本和响应时间
- 优化提示词长度，避免不必要的 Token 消耗
- 理解模型为何对某些文本理解不准确（Token 边界切割导致语义断裂）

::: tip 提示
可以使用各平台提供的 Tokenizer 工具（如 OpenAI Tokenizer、Tiktoken）精确计算提示词的 Token 数量，提前评估成本。
:::

### 上下文窗口

**上下文窗口（Context Window）**是模型在一次交互中能够处理的 Token 总量上限，包含输入提示词和生成输出的总和。

当前主流模型的上下文窗口从 8K 到 200K Token 不等。理解上下文窗口的限制对提示词设计至关重要：

- **提示词长度与输出长度的权衡**：上下文窗口 = 输入 Token + 输出 Token。输入越长，留给输出的空间越小
- **"中间丢失"现象（Lost in the Middle）**：研究表明，模型对上下文开头和结尾的信息关注度更高，中间部分容易被忽略。关键指令应放在提示词的开头或结尾
- **长文本处理策略**：当内容超出上下文窗口时，需要采用分块处理（Chunking）、摘要压缩（Summarization）或检索增强（Retrieval Augmented）等策略

::: warning 注意
不要假设模型能"记住"超长上下文中的所有细节。对于关键约束和格式要求，务必放在提示词的显眼位置（开头或结尾），并在必要时重复强调。
:::

### 温度参数

**温度（Temperature）**是控制模型输出随机性的核心参数，取值范围通常为 0 到 2。

温度参数的作用机制是调整模型生成下一个 Token 时的概率分布：

- **低温度（0-0.3）**：概率分布更尖锐，模型倾向于选择概率最高的 Token。输出更确定、更一致，适合代码生成、数据提取、格式转换等需要精确结果的场景
- **中温度（0.4-0.6）**：平衡确定性与创造性，适合大多数通用任务
- **高温度（0.7-1.0）**：概率分布更平坦，模型更可能选择低概率 Token。输出更多样、更有创意，适合头脑风暴、创意写作、故事生成等需要发散思维的场景

| 温度值 | 输出特征               | 适用场景           |
| ------ | ---------------------- | ------------------ |
| 0      | 完全确定，每次输出相同 | 数据校验、格式转换 |
| 0.2    | 高度确定，极少变化     | 代码生成、技术文档 |
| 0.5    | 适度变化               | 摘要、翻译、问答   |
| 0.8    | 明显变化               | 创意写作、广告文案 |
| 1.0    | 高度随机               | 头脑风暴、灵感激发 |

::: tip 提示
调试提示词时建议先将温度设为 0，确保指令本身有效后再逐步提高温度以获得更自然的输出。
:::

## Zero-shot Prompting

### 概念

**Zero-shot Prompting（零样本提示）**是指不提供任何示例，直接向模型下达指令的提示方式。模型完全依赖预训练阶段学到的知识来完成任务。

适用场景包括：

- **简单任务**：文本分类、情感判断、语言识别等模型已经充分学习的任务
- **常识问答**：基于广泛知识的问答，如历史事件、科学常识
- **格式转换**：大小写转换、日期格式标准化等规则明确的操作

### 示例对比

```text
# 不好的提示词
"总结一下这篇文章"

# 好的提示词
"请用 3 个要点总结以下文章，每个要点不超过 20 字，重点突出技术实现细节"
```

不好的提示词问题在于：没有指定输出数量、长度限制和重点方向，模型可能输出过长或过短的内容，也可能关注错误的方面。

好的提示词明确了四个要素：

- **任务类型**：总结
- **输出数量**：3 个要点
- **长度约束**：每个要点不超过 20 字
- **关注重点**：技术实现细节

### 最佳实践

编写 Zero-shot 提示词时遵循以下原则：

- **指令要具体明确**：使用可衡量的描述，如"列出 5 个优点"而非"列出一些优点"
- **包含完整要素**：任务类型、输入格式、输出格式、约束条件缺一不可
- **避免模糊词汇**："简要"、"详细"、"适当"等词汇在不同人理解中差异很大，应替换为具体的数字或范围
- **使用肯定句**：明确告诉模型"要做什么"比"不要做什么"更有效

::: info 信息
Zero-shot 是最高效的提示方式，应优先尝试。只有在 Zero-shot 效果不满足需求时，才考虑添加示例（Few-shot）或使用更复杂的提示策略。
:::

## Few-shot Prompting

### 概念

**Few-shot Prompting（少样本提示）**通过在提示词中提供 1 到 5 个输入输出示例，让模型学习期望的输出格式、风格和模式。这种方式利用了模型的上下文学习（In-context Learning）能力。

Few-shot 的核心价值在于：

- **格式示范**：通过示例展示精确的输出格式，比文字描述更直观
- **风格引导**：让模型理解期望的语言风格、语气和专业程度
- **边界定义**：通过正反面示例告诉模型什么该做、什么不该做

### 示例设计原则

设计 Few-shot 示例时遵循三个关键原则：

- **代表性**：示例应覆盖任务中的典型场景，让模型学到通用模式而非特例
- **多样性**：展示不同情况下的处理方式，避免模型过度拟合单一模式
- **格式一致**：示例的输入输出格式必须与目标任务完全一致，否则会造成混淆

### 效果对比

| 示例数量 | 适用场景     | 优点                 | 缺点               |
| -------- | ------------ | -------------------- | ------------------ |
| 1-shot   | 简单格式转换 | 节省 Token，快速验证 | 学习效果有限       |
| 3-shot   | 大多数任务   | 平衡效果与成本       | 需要精心设计示例   |
| 5-shot   | 复杂模式学习 | 学习效果好，覆盖全面 | 占用较多上下文窗口 |

::: tip 提示
示例数量并非越多越好。当示例超过 5 个时，边际效益显著递减，且可能挤占上下文窗口。优先保证示例质量而非数量。
:::

### 完整示例

```text
将以下口语化表达转换为正式书面语：

输入：这个东西不太好使
输出：该产品的使用体验有待改善

输入：我觉得这个方案不太行
输出：该方案在可行性方面存在一定疑虑

输入：这个 bug 我搞不定
输出：
```

在这个示例中，模型通过前两个示例学到了：

- 将口语化词汇替换为书面语（"不好使" → "使用体验有待改善"）
- 将主观感受转化为客观描述（"我觉得" → 隐含的客观陈述）
- 保持简洁专业的语气

当模型看到第三个输入时，会基于已学到的模式生成类似"该问题的解决存在一定难度"的输出。

## 角色扮演提示（Role Prompting）

### 原理

**角色扮演提示（Role Prompting）**通过为模型指定专业角色，激活其在特定领域的知识和能力。当模型被赋予某个角色时，它会倾向于使用该角色的专业术语、思维方式和表达风格。

角色设定的作用机制：

- **知识激活**：引导模型从预训练数据中检索与角色相关的知识子集
- **风格对齐**：自动调整语言风格以匹配角色特征（如工程师的严谨、营销人员的感染力）
- **视角聚焦**：让模型从特定视角分析问题，避免泛泛而谈

### 模板

```text
你是一位{角色}，擅长{技能}。请{任务描述}。
```

### 示例

```text
你是一位资深的前端工程师，精通 React 和 TypeScript。请审查以下代码，指出潜在的性能问题并给出优化建议。
```

这个提示词中：

- **角色**：资深前端工程师
- **技能**：React 和 TypeScript
- **任务**：代码审查，指出性能问题并给出优化建议

相比"请审查这段代码"，角色设定让模型更可能关注前端特有的性能问题（如不必要的重渲染、状态管理不当等），而非泛泛的代码风格问题。

### 多角色协作示例

对于复杂任务，可以让模型依次扮演多个角色，从不同视角分析问题：

```text
你将依次扮演三个角色分析以下需求：

1. 产品经理：分析用户需求和业务价值
2. 架构师：设计技术方案和系统架构
3. 工程师：评估实现难度和开发周期

请依次以三个角色的视角分析以下需求：开发一个支持多人实时协作的在线文档编辑器
```

这种多角色协作方式能够：

- 获得更全面的分析视角
- 发现单一角色可能忽略的问题
- 生成结构化的多维度输出

::: warning 注意
多角色提示词会显著增加 Token 消耗和响应时间。仅在需要多维度分析时使用，简单任务使用单一角色即可。
:::

## 提示词基本公式

### 标准公式

一个完整的提示词应包含以下五个核心要素：

**角色 + 任务 + 约束 + 格式 + 示例**

这五个要素构成了提示词的基本骨架，缺一不可：

- **角色**：定义模型的身份和专业领域
- **任务**：明确需要完成的具体工作
- **约束**：设定边界条件和限制规则
- **格式**：指定期望的输出结构
- **示例**：提供输入输出的参考样例

### 完整模板

```markdown
# 角色

{role_description}

# 任务

{task_description}

# 约束

{constraints}

# 输出格式

{output_format}

# 示例

{examples}

# 输入

{user_input}
```

使用 Markdown 标题分隔各部分，让模型清晰识别不同指令的边界。这种结构化写法也便于提示词的版本管理和团队协作。

### 实战示例

```text
# 角色
你是一位经验丰富的技术文档工程师

# 任务
将以下技术文档转换为用户友好的分步操作说明

# 约束
- 每步操作不超过 50 字
- 避免使用技术术语
- 使用动作动词开头

# 输出格式
1. [第一步操作]
2. [第二步操作]
...

# 输入
{技术文档内容}
```

这个模板的优势在于：

- 结构清晰，模型容易理解各部分的含义
- 约束具体可衡量（50 字、动作动词开头）
- 输出格式明确，便于后续程序化处理

### 常见错误与修正

| 错误                               | 修正                                            |
| ---------------------------------- | ----------------------------------------------- |
| 指令模糊："简要总结"               | 使用具体描述："用 3 句话总结，每句不超过 30 字" |
| 缺少约束：没有格式要求             | 明确约束：指定字数、格式、风格、语言            |
| 没有示例：模型不理解期望输出       | 添加 1 到 3 个代表性示例                        |
| 角色缺失：直接下达任务             | 先设定角色再描述任务                            |
| 输入未隔离：用户输入与指令混在一起 | 使用分隔符（如引号、横线）隔离输入内容          |

::: warning 注意
当用户输入可能包含指令性内容时，务必使用分隔符将其与系统指令隔离，防止**提示词注入（Prompt Injection）**攻击。推荐使用三重引号 `"""` 或 `---` 作为分隔符。
:::
