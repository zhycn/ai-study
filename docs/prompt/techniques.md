# 技巧篇：常用提示词模式

提示词工程（Prompt Engineering）不仅是理论，更是实践。掌握常用的提示词模式，能够显著提升大语言模型（LLM）的输出质量与稳定性。本章将系统讲解结构化输出、分隔符使用、负向提示、上下文管理、多模型适配以及调试迭代等核心技巧，帮助你在实际项目中高效运用提示词。

## 结构化输出

结构化输出是指**明确要求模型按照特定格式生成内容**，使输出结果可以直接被程序解析或人工快速阅读。这是提示词工程中最实用的技巧之一。

### JSON 格式约束

当需要将模型输出接入下游系统（如 API、数据库、前端组件）时，JSON 是最常用的格式。关键在于**提供完整的 JSON Schema 示例**，让模型严格遵循字段名称和类型。

```text
请以 JSON 格式输出，包含以下字段：
{
  "summary": "摘要内容",
  "key_points": ["要点1", "要点2", "要点3"],
  "sentiment": "positive/negative/neutral"
}

要求：
- 仅输出 JSON，不要添加任何额外文字
- key_points 必须恰好包含 3 个字符串
- sentiment 只能从三个选项中选择一个
```

::: tip 提示
在 JSON 提示词末尾添加"仅输出 JSON，不要添加任何额外文字"，可以有效避免模型输出 ```json 代码块标记或解释性文字。
:::

### Markdown 格式约束

当输出面向人类读者时，Markdown 格式更为友好。通过**预定义标题层级和段落结构**，可以确保每次生成的内容风格一致。

```text
请按以下 Markdown 格式输出：
## 概述
[100 字以内的概述]

## 关键要点
- 要点 1
- 要点 2
- 要点 3

## 建议
[3 条具体可行的建议]
```

这种方式的优点在于**结构清晰、可读性强**，适合生成报告、文档、分析文章等场景。

### 表格输出技巧

表格适合**对比分析、分类整理、数据展示**等场景。提示词中需要明确列名，必要时给出示例行。

```text
请以 Markdown 表格形式输出，包含以下列：
| 功能 | 描述 | 适用场景 | 示例 |

要求：
- 至少列出 5 行数据
- 描述不超过 30 字
- 示例使用具体产品或工具名称
```

::: info 信息
表格输出时，模型偶尔会在单元格内使用换行导致格式错乱。可以在提示词中补充"单元格内容不要换行"来规避。
:::

### 最佳实践

- **明确指定输出格式**：不要说"请总结"，而要说"请用 3 个要点总结，每个要点不超过 20 字"
- **提供格式示例**：给出一个完整的输出样例，比长篇描述更有效
- **使用强调词**：加入"必须"、"严格"、"仅"等词汇强化约束
- **验证输出**：对 JSON 等结构化输出，建议在代码层做格式校验和异常处理

## 分隔符使用

分隔符是提示词工程中的**基础防御机制**，用于清晰区分系统指令与用户输入内容，防止模型混淆不同部分。

### 为什么需要分隔符

大语言模型会将整段输入视为连续文本。如果不使用分隔符，模型可能**无法区分哪些是指令、哪些是待处理的内容**，导致以下问题：

- 用户输入中包含类似指令的文字时，模型可能误执行
- 长文本中关键指令被淹没，模型忽略约束条件
- 提示词结构混乱，输出质量不稳定

### 常用分隔符

| 分隔符       | 适用场景     | 示例                    |
| ------------ | ------------ | ----------------------- |
| `"""`        | 包裹长文本   | `"""{文章内容}"""`      |
| `---`        | 分隔章节     | 视觉分隔不同部分        |
| `###`        | 标题层级     | Markdown 标题分隔       |
| `<tag>`      | XML 风格标签 | `<input>{内容}</input>` |
| `[[` 和 `]]` | 变量占位符   | `[[用户输入]]`          |

::: warning 注意
当用户输入可能包含指令性内容时，务必使用分隔符隔离，防止**提示词注入（Prompt Injection）**。这是生产环境中最常见的安全风险。
:::

### 完整示例

```text
请总结以下文章的核心要点：

"""
{文章内容}
"""

要求：
- 提取 3 个要点
- 每个要点不超过 30 字
- 按重要性排序
```

XML 风格标签在复杂场景中更为推荐，因为**标签具有语义性**，模型更容易理解各部分的含义：

```text
请根据以下材料回答问题。

<材料>
{文章内容}
</材料>

<问题>
{用户问题}
</问题>

要求：
- 仅基于材料内容回答
- 如果材料中没有相关信息，请明确说明
```

### 最佳实践

- **优先使用 XML 标签**：如 `<context>`、`<input>`、`<instructions>`，语义清晰
- **嵌套使用**：复杂提示词可以分层嵌套，如 `<task><input>...</input><constraints>...</constraints></task>`
- **一致性**：在整个项目中保持分隔符风格统一，便于维护和团队协作

## 负向提示（Negative Prompting）

负向提示是指**明确告诉模型不要做什么**，通过划定边界来约束输出范围。它与正向提示（告诉模型要做什么）配合使用，效果最佳。

### 概念

大语言模型具有很强的"创造力"，但这也意味着它可能**添加原文没有的内容、编造数据或偏离主题**。负向提示通过显式禁止某些行为，减少模型的"过度发挥"。

### 示例

```text
请总结以下技术文章：

"""
{文章内容}
"""

请注意：
- 不要编造数据或引用不存在的来源
- 不要添加原文没有的观点
- 不要使用专业术语而不加解释
- 不要输出与任务无关的内容
- 不要改变原文的立场或结论
```

### 适用场景

| 场景     | 负向提示示例                 |
| -------- | ---------------------------- |
| 内容摘要 | 不要添加原文没有的观点       |
| 代码生成 | 不要使用已废弃的 API         |
| 翻译任务 | 不要意译，保持原文语气       |
| 数据分析 | 不要推测数据趋势，仅描述事实 |
| 客服回复 | 不要承诺具体解决时间         |

::: tip 提示
负向提示的数量建议控制在 3-5 条。过多负向约束可能让模型"不知所措"，反而影响输出质量。
:::

### 最佳实践

- **具体而非笼统**：用"不要使用 2020 年之前的数据"替代"不要使用旧数据"
- **与正向提示搭配**：先说"要做什么"，再说"不要做什么"
- **优先级排序**：最重要的约束放在最前面
- **避免矛盾**：确保负向提示与正向指令不冲突

## 上下文管理

上下文管理是**在有限上下文窗口（Context Window）内高效组织信息**的技巧。随着对话轮次增加，合理管理上下文变得尤为重要。

### 指令遗忘问题

大语言模型的上下文窗口虽然越来越大（从 4K 到 200K 甚至 1M token），但仍存在**指令遗忘**现象：

- 长对话中模型可能遗忘早期的系统指令
- 关键约束在中间轮次被"稀释"
- 上下文接近上限时，模型对首尾内容的关注度高于中间内容（**首尾偏好效应**）

### 上下文压缩策略

| 策略     | 方法                             | 适用场景     |
| -------- | -------------------------------- | ------------ |
| 定期总结 | 每 5-10 轮对话总结一次要点       | 长对话场景   |
| 重述指令 | 在新任务开始时重述关键约束       | 多任务切换   |
| 强调词   | 使用"记住"、"始终"、"全程"等词汇 | 全局约束     |
| 分段处理 | 将长文本拆分为多个独立任务       | 超长文档处理 |

### 示例

```text
[系统指令 - 请在整个对话中始终遵守]
- 使用中文回答
- 代码必须包含注释
- 技术术语首次出现时标注英文

[当前任务]
请解释以下概念：{概念名称}
```

对于超长对话，可以采用**上下文刷新**策略：

```text
以下是我们之前讨论的要点总结：

<对话摘要>
1. 项目目标：构建一个 RAG 系统
2. 技术栈：LangChain + PostgreSQL
3. 当前阶段：正在设计检索策略
</对话摘要>

基于以上背景，请继续回答：{新问题}
```

::: warning 注意
不要依赖模型"记住"所有历史对话。关键信息应主动总结并重新注入上下文。
:::

### 最佳实践

- **关键指令放首尾**：利用首尾偏好效应，将最重要的约束放在提示词开头或末尾
- **定期压缩上下文**：对话超过 10 轮时，主动总结并重置上下文
- **使用锚点标记**：用 `[重要]`、`[必须遵守]` 等标记增强关键指令的可见性
- **控制总长度**：提示词 + 示例 + 输入的总长度建议不超过上下文窗口的 70%

## 多模型适配

不同的大语言模型在架构、训练数据和指令微调方式上存在差异，**同一提示词在不同模型上的表现可能截然不同**。

### 主流模型特点对比

| 模型       | 提示词策略                        | 优势                             | 注意事项                                 |
| ---------- | --------------------------------- | -------------------------------- | ---------------------------------------- |
| GPT-4o     | 支持复杂指令，多层嵌套结构        | 指令遵循能力强，输出稳定         | 避免过度复杂的嵌套，保持逻辑清晰         |
| Claude     | 擅长长文本，详细上下文 + 复杂任务 | 长上下文窗口（200K），推理能力强 | 充分利用长文本优势，可放入完整参考文档   |
| Gemini     | 多模态能力强，结合图像提示        | 原生支持图像、音频、视频输入     | 可利用多模态输入增强提示效果             |
| 开源小模型 | 指令需简单明确，多给示例          | 可本地部署，数据隐私好           | 需要更多 Few-shot 示例引导，指令不宜过长 |

### 适配建议

- **针对目标模型单独优化**：不要假设提示词在所有模型上效果一致
- **建立模型特定的提示词模板库**：为每个常用模型维护一套经过验证的提示词
- **利用模型特性**：如 Claude 的长上下文、Gemini 的多模态、GPT 的函数调用
- **跨模型测试**：关键业务场景应在多个模型上验证提示词效果

```text
# GPT-4o 适配版本（结构化指令）
你是一个代码审查助手。请按以下步骤执行：
1. 分析代码结构
2. 识别潜在问题
3. 给出优化建议
4. 输出 JSON 格式报告

# Claude 适配版本（自然语言描述）
你是一位经验丰富的代码审查专家。请仔细阅读以下代码，
从可读性、性能、安全性三个维度进行全面分析，
并给出具体的改进建议。你可以参考以下审查标准...
（附上详细的审查标准文档）
```

::: tip 提示
如果业务需要跨模型兼容，建议以**能力最弱的模型为基准**编写提示词，确保在所有目标模型上都能正常工作。
:::

### 最佳实践

- **抽象提示词层**：将提示词模板与模型调用逻辑分离，便于独立优化
- **版本化管理**：为每个模型的提示词维护版本号，记录变更历史
- **自动化评估**：建立跨模型的提示词效果评估流水线
- **关注模型更新**：模型版本升级后，重新验证提示词效果

## 调试与迭代

提示词优化是一个**持续迭代的过程**。通过系统化的调试方法，可以快速定位问题并提升输出质量。

### 隔离变量法

调试提示词时，**每次只修改一个变量**，以便准确评估该修改对输出的影响。

| 变量     | 修改示例                          |
| -------- | --------------------------------- |
| 角色设定 | 从"助手"改为"资深专家"            |
| 指令措辞 | 从"请总结"改为"请用 3 个要点总结" |
| 示例数量 | 从 1 个 Few-shot 增加到 3 个      |
| 输出格式 | 从自由文本改为 JSON 格式          |
| 分隔符   | 从 `"""` 改为 `<input>` 标签      |

### 检查清单

在每次修改提示词后，逐项检查：

- [ ] 指令是否有歧义？是否存在多种理解方式？
- [ ] 输出格式是否明确？模型能否准确遵循？
- [ ] 是否需要添加示例？Few-shot 是否覆盖典型场景？
- [ ] 任务是否可以进一步简化？是否过于复杂？
- [ ] 负向约束是否充分？是否遗漏了常见的错误输出？
- [ ] 分隔符是否正确使用了？用户输入是否被有效隔离？

### 调试流程

1. **单样本测试**：用 3-5 个典型输入测试提示词效果，观察输出是否符合预期
2. **边界测试**：用极端或异常输入测试鲁棒性，如空输入、超长输入、包含特殊字符的输入
3. **对比测试**：对比不同提示词版本的输出质量，记录每次修改的改进点
4. **收集反馈**：让目标用户或领域专家评估输出，收集定性反馈
5. **回归测试**：建立测试集，每次修改后批量验证，确保不引入新的问题

```text
# 调试记录示例
版本 v1.0 -> v1.1
修改：将"请总结"改为"请用 3 个要点总结，每个不超过 20 字"
效果：输出长度稳定性提升 80%，但部分要点信息丢失
下一步：调整为"每个要点 20-30 字"，平衡信息量与简洁性
```

::: info 信息
建议将提示词及其测试用例纳入版本控制系统（如 Git），便于追踪变更历史和回滚到稳定版本。
:::

### 最佳实践

- **建立测试集**：覆盖典型场景、边界情况和异常输入
- **量化评估**：定义可衡量的指标（准确率、格式合规率、响应时间等）
- **记录调试日志**：每次修改都记录变更内容和效果对比
- **小步快跑**：频繁小幅度调整优于一次性大改

## 与其他概念的关系

- [温度](/glossary/temperature)（Temperature）参数影响输出的随机性，调试时可以适当调高以观察模型的不同回答，或调低以确保输出稳定
- [上下文窗口](/glossary/context-window)（Context Window）限制了提示词和示例的总长度，需要合理分配各部分的 token 占比
- [提示词注入](/glossary/prompt-injection)（Prompt Injection）是需要通过分隔符防御的安全风险，生产环境必须重视
- [少样本提示](/glossary/prompt-engineering)（Few-shot Prompting）提供示例引导模型输出，是结构化输出的重要辅助手段
- [思维链](/prompt/advanced)（Chain of Thought）在进阶篇中详细讲解，与负向提示配合可以提升推理任务的准确性
