# 模式库：可复用的提示词模板集合

## 如何使用本模式库

- 每个模板包含场景描述、模板代码、使用示例和注意事项
- 模板中的 `{变量}` 需要替换为实际内容
- 可以根据实际需求组合多个模板
- 建议先阅读[基础篇](/prompt/fundamentals)和[技巧篇](/prompt/techniques)了解基本原理

:::tip 提示
模板是**提示词工程（Prompt Engineering）**中提高效率和一致性的有效手段。建立自己的模板库可以大幅提升与大语言模型（LLM）的交互质量。
:::

## 文本处理模式

### 摘要生成模板

**场景**：将长文本压缩为核心要点

```text
请用 {num_points} 个要点总结以下内容，每个要点不超过 {max_chars} 字：

"""
{content}
"""

要求：
- 重点突出{focus_area}
- 使用简洁的语言
- 保持原文的核心观点
```

**示例**：

```text
请用 3 个要点总结以下内容，每个要点不超过 30 字：

"""
Transformer 架构是 2017 年提出的深度学习模型，核心创新是自注意力机制（Self-Attention Mechanism），
它能够并行处理序列数据，大幅提升了训练效率。该架构已被广泛应用于自然语言处理（NLP）、
计算机视觉（CV）等领域，成为现代 AI 系统的基石。
"""

要求：
- 重点突出技术创新
- 使用简洁的语言
- 保持原文的核心观点
```

**注意事项**：

- 要点数量建议 3-5 个，过多会导致信息稀释
- 对于技术文档，可以要求保留关键数据
- 如果原文结构复杂，建议先分段再摘要

### 风格转换模板

**场景**：将文本从一种风格转换为另一种

```text
请将以下文本从{source_style}转换为{target_style}：

"""
{text}
"""

转换要求：
- 保持原意不变
- 调整语气和用词
- 符合{target_style}的表达习惯
```

**风格选项**：

| 源风格   | 目标风格   | 示例                                      |
| -------- | ---------- | ----------------------------------------- |
| 口语化   | 正式书面语 | "这东西不好用" → "该产品体验有待改善"     |
| 技术文档 | 用户指南   | "执行 kubectl get pods" → "查看 Pod 状态" |
| 中文     | 英文       | 保持专业术语准确                          |

**示例**：

```text
请将以下文本从口语化转换为正式书面语：

"""
这个功能真的很难用，每次都要点好多下才能找到，建议改一下。
"""

转换要求：
- 保持原意不变
- 调整语气和用词
- 符合正式书面语的表达习惯
```

**注意事项**：

- 明确定义源风格和目标风格，避免模糊描述
- 对于专业领域文本，需要额外说明术语处理方式
- 转换后建议人工复核，确保语义未发生偏移

### 翻译优化模板

**场景**：高质量翻译，保留专业术语

```text
请将以下内容从{source_lang}翻译为{target_lang}：

"""
{text}
"""

翻译要求：
- 专业术语保留原文并在括号中标注
- 符合{target_lang}的表达习惯
- 技术描述必须准确
- 遇到不确定的术语请标注"[待确认]"
```

**示例**：

```text
请将以下内容从英文翻译为中文：

"""
The model uses a multi-head attention mechanism to capture dependencies
across different representation subspaces. Each head learns different
attention patterns, which are then concatenated and linearly projected.
"""

翻译要求：
- 专业术语保留原文并在括号中标注
- 符合中文的表达习惯
- 技术描述必须准确
- 遇到不确定的术语请标注"[待确认]"
```

**注意事项**：

- 对于技术文档，建议提供术语表（Glossary）作为参考
- 长文本建议分段翻译，保持上下文一致性
- 翻译完成后建议进行回译（Back-Translation）验证

## 代码辅助模式

### 代码审查模板

**场景**：审查代码质量和潜在问题

````text
你是一位资深的{language}工程师，请审查以下代码：

```{language}
{code}
````

请从以下维度进行审查：

1. **功能正确性**：逻辑是否有漏洞
2. **性能问题**：是否有可优化的空间
3. **代码规范**：是否符合最佳实践
4. **安全隐患**：是否有安全风险
5. **可维护性**：是否易于理解和扩展

输出格式：
| 问题类型 | 位置 | 描述 | 建议 |
|---------|------|------|------|

````

**示例**：

```text
你是一位资深的 Python 工程师，请审查以下代码：

```python
def get_user_data(user_id):
    conn = sqlite3.connect('users.db')
    cursor = conn.cursor()
    query = f"SELECT * FROM users WHERE id = {user_id}"
    cursor.execute(query)
    return cursor.fetchone()
````

请从以下维度进行审查：

1. **功能正确性**：逻辑是否有漏洞
2. **性能问题**：是否有可优化的空间
3. **代码规范**：是否符合最佳实践
4. **安全隐患**：是否有安全风险
5. **可维护性**：是否易于理解和扩展

输出格式：
| 问题类型 | 位置 | 描述 | 建议 |
|---------|------|------|------|

````

**注意事项**：

- 提供完整的代码上下文，避免片段审查
- 明确指出代码的语言和版本
- 对于安全敏感代码，建议增加**静态应用安全测试（SAST）**检查

:::warning 安全警告
AI 代码审查不能替代人工审查和安全测试工具。对于生产环境代码，必须结合专业安全审计。
:::

### Bug 修复模板

**场景**：定位并修复代码中的 Bug

```text
请帮我定位并修复以下代码中的 Bug：

```{language}
{code}
````

错误信息：

```
{error_message}
```

请按以下步骤：

1. 分析错误原因
2. 定位问题代码行
3. 给出修复方案
4. 解释修复原理
5. 提供修复后的完整代码

````

**示例**：

```text
请帮我定位并修复以下代码中的 Bug：

```python
def calculate_average(numbers):
    total = 0
    for i in range(len(numbers) + 1):
        total += numbers[i]
    return total / len(numbers)
````

错误信息：

```
IndexError: list index out of range
```

请按以下步骤：

1. 分析错误原因
2. 定位问题代码行
3. 给出修复方案
4. 解释修复原理
5. 提供修复后的完整代码

````

**注意事项**：

- 提供完整的错误堆栈（Stack Trace）
- 说明触发 Bug 的输入数据
- 修复后建议补充对应的**单元测试（Unit Test）**

### 代码解释模板

**场景**：解释复杂代码的功能和原理

```text
请解释以下代码的功能和实现原理：

```{language}
{code}
````

解释要求：

- 用通俗的语言描述功能
- 逐行或逐段解释关键逻辑
- 说明使用的设计模式（如有）
- 指出可能的优化空间

````

**示例**：

```text
请解释以下代码的功能和实现原理：

```python
def fibonacci(n, memo={}):
    if n in memo:
        return memo[n]
    if n <= 2:
        return 1
    memo[n] = fibonacci(n - 1, memo) + fibonacci(n - 2, memo)
    return memo[n]
````

解释要求：

- 用通俗的语言描述功能
- 逐行或逐段解释关键逻辑
- 说明使用的设计模式（如有）
- 指出可能的优化空间

````

**注意事项**：

- 指定目标读者的技术水平（初学者/中级/高级）
- 对于算法代码，建议要求分析**时间复杂度（Time Complexity）**和**空间复杂度（Space Complexity）**
- 可以要求生成可视化流程图辅助理解

### 测试生成模板

**场景**：为代码生成单元测试

```text
请为以下代码生成单元测试：

```{language}
{code}
````

要求：

- 使用{test_framework}测试框架
- 覆盖正常场景和边界场景
- 包含至少{test_count}个测试用例
- 每个测试用例包含注释说明

````

**示例**：

```text
请为以下代码生成单元测试：

```python
def is_palindrome(s: str) -> bool:
    s = s.lower().replace(' ', '')
    return s == s[::-1]
````

要求：

- 使用 pytest 测试框架
- 覆盖正常场景和边界场景
- 包含至少 5 个测试用例
- 每个测试用例包含注释说明

````

**注意事项**：

- 明确测试框架版本
- 要求覆盖**边界值（Boundary Value）**和**异常路径（Edge Case）**
- 建议同时生成**测试覆盖率（Test Coverage）**报告命令

```text

{data}

分析要求：
1. 描述数据的基本特征（均值、中位数、分布等）
2. 识别异常值或趋势
3. 得出 3-5 条关键洞察
4. 给出可行动的建议
````

**示例**：

```text
请分析以下数据并给出洞察：

2月  | 12000 | 42%   | 55
3月  | 15000 | 38%   | 60
4月  | 18000 | 35%   | 65
5月  | 25000 | 30%   | 70

分析要求：
1. 描述数据的基本特征（均值、中位数、分布等）
2. 识别异常值或趋势
3. 得出 3-5 条关键洞察
4. 给出可行动的建议
```

**注意事项**：

- 提供数据的背景信息和采集方式
- 对于时间序列数据，说明时间粒度
- AI 分析结果需要结合业务实际验证

:::tip 提示

```text
  "intent": "类别",
  "confidence": 0.0-1.0,
  "entities": ["提取的关键实体"],
  "response_strategy": "建议的回复策略"
}
```

**注意事项**：

- 根据业务场景自定义意图类别
- 对于低置信度（Confidence < 0.7）的结果，建议转人工处理
- 可以要求同时输出**情感分析（Sentiment Analysis）**结果

---

### 多轮对话管理模板

**场景**：管理多轮对话的上下文

```text
你是一位对话助手，当前对话状态：

对话历史：
{conversation_history}

当前用户输入：
{current_input}

请：
1. 理解当前对话的上下文
2. 识别用户意图
3. 生成合适的回复
4. 更新对话状态

注意：
- 保持对话的连贯性
- 如果用户切换话题，及时跟进
- 记住用户之前提到的关键信息
```

**示例**：

```text
你是一位对话助手，当前对话状态：

对话历史：
用户：我想订一张明天北京到上海的机票
助手：好的，请问您希望什么时间段出发？
用户：上午的航班

当前用户输入：
最好是国航或者东航的

请：
1. 理解当前对话的上下文
2. 识别用户意图
3. 生成合适的回复
4. 更新对话状态

注意：
- 保持对话的连贯性
- 如果用户切换话题，及时跟进
- 记住用户之前提到的关键信息
```

**注意事项**：

- 对话历史需要合理截断，避免超出上下文窗口（Context Window）
- 对于关键信息（如订单号、个人信息），建议显式确认
- 可以要求输出结构化的对话状态，便于系统处理

## Agent 协作模式

### 任务分解模板

**场景**：将复杂任务拆分为子任务

```text
请将以下复杂任务拆分为可执行的子任务：

任务：{complex_task}

拆分要求：
1. 每个子任务应该是独立的
2. 明确每个子任务的输入和输出
3. 标注子任务之间的依赖关系
4. 建议执行顺序

输出格式：
| 序号 | 子任务 | 输入 | 输出 | 依赖 |
|------|--------|------|------|------|
```

**示例**：

```text
请将以下复杂任务拆分为可执行的子任务：

任务：开发一个用户反馈分析系统，能够从用户评论中自动提取情感倾向和关键主题

拆分要求：
1. 每个子任务应该是独立的
2. 明确每个子任务的输入和输出
3. 标注子任务之间的依赖关系
4. 建议执行顺序

输出格式：
| 序号 | 子任务 | 输入 | 输出 | 依赖 |
|------|--------|------|------|------|
```

**注意事项**：

- 任务描述越清晰，拆分结果越合理
- 对于强依赖的子任务，建议增加验证步骤
- 可以参考**工作流（Workflow）**模式组织子任务执行

### 工具选择模板

**场景**：根据任务需求选择合适的工具

```text
我需要完成以下任务：
{task_description}

可用工具列表：
{available_tools}

请：
1. 分析任务需求
2. 推荐最适合的工具
3. 说明选择理由
4. 给出工具组合建议（如需要多个工具）
```

**示例**：

```text
我需要完成以下任务：
分析最近 30 天的用户行为数据，生成可视化报告

可用工具列表：
- Python Pandas：数据处理
- Python Matplotlib：基础可视化
- Python Seaborn：统计可视化
- ECharts：交互式图表
- Tableau：商业智能分析

请：
1. 分析任务需求
2. 推荐最适合的工具
3. 说明选择理由
4. 给出工具组合建议（如需要多个工具）
```

**注意事项**：

- 提供工具的详细能力描述
- 说明团队的技术栈偏好
- 考虑学习成本和部署环境限制

### 结果汇总模板

**场景**：汇总多个 Agent 或步骤的输出

```text
请汇总以下任务的执行结果：

任务目标：{task_objective}

各步骤输出：
{step_outputs}

汇总要求：
1. 总结整体进展
2. 列出关键成果
3. 指出存在的问题
4. 给出下一步建议
```

**示例**：

```text
请汇总以下任务的执行结果：

任务目标：完成竞品分析报告

各步骤输出：
步骤 1（数据收集）：已收集 5 个竞品的产品信息、定价策略、用户评价
步骤 2（功能对比）：完成功能矩阵对比表，发现我方在 AI 功能上有优势
步骤 3（用户调研）：完成 100 份用户问卷，60% 用户认为我方 UI 更友好
步骤 4（市场趋势）：行业年增长 25%，AI 集成是主要趋势

汇总要求：
1. 总结整体进展
2. 列出关键成果
3. 指出存在的问题
4. 给出下一步建议
```

**注意事项**：

- 提供各步骤的完整输出，避免信息丢失
- 对于冲突的结果，建议标注并分析原因
- 汇总结果可以作为**决策支持（Decision Support）**的输入

### 如何选择合适的模板

| 任务类型 | 推荐模板 | 参考文档                       |
| -------- | -------- | ------------------------------ |
| 长文总结 | 摘要生成 | [基础篇](/prompt/fundamentals) |

| 数据洞察 | 数据解读 | [进阶篇](/prompt/advanced)     |
| 营销文案 | 文案撰写 | [技巧篇](/prompt/techniques)   |
| 客服系统 | 客服回复 | [工程篇](/prompt/engineering)  |
| 复杂任务 | 任务分解 | [进阶篇](/prompt/advanced)     |

### 自定义模板的方法

1. **复制基础模板**：从本模式库选择合适的模板
2. **调整变量**：根据实际需求修改变量
3. **添加约束**：增加特定场景的约束条件
4. **测试验证**：用 3-5 个样本测试效果
5. **迭代优化**：根据测试结果调整

:::info 提示
建议建立自己的模板库，将验证有效的模板分类存储，并记录使用场景和效果反馈。
:::

### 模板组合技巧

- **串联组合**：将多个模板按顺序执行（见[进阶篇 - 提示词链](/prompt/advanced)）
- **并行组合**：同一输入使用多个模板，汇总结果
- **条件组合**：根据输入类型选择不同的模板

**示例：串联组合**

```text
# 第一步：使用摘要生成模板
输入：长篇文章 → 输出：核心要点

# 第二步：使用风格转换模板
输入：核心要点 → 输出：正式报告风格

# 第三步：使用格式转换模板
输入：正式报告 → 输出：Markdown 格式
```
